# ═══════════════════════════════════════════════════════════════════════════
# PROMPTING STRATEGY DESCRIPTIONS
# ═══════════════════════════════════════════════════════════════════════════
#
# NAIVE
#   Issue body: a single bullet point stating only the bare task.
#   No mention of tests, edge cases, artifacts, or implementation details.
#   Example: "Add factorial to the calculator."
#   Goal: measure what Claude produces with the least possible guidance.
#
# STRUCTURED
#   Issue body: 2–3 bullet points.
#   Names the feature precisely, explicitly asks for tests to be updated,
#   and mentions keeping development artifacts consistent.
#   No implementation hints or edge-case enumeration.
#   Example:
#     - Add factorial as a supported calculator operation.
#     - Make sure the calculator can correctly compute factorial values for
#       valid inputs and update the tests accordingly.
#
# EXPERT-GUIDED
#   Issue body: 3+ bullet points with edge cases, constraints, and examples.
#   Specifies boundary conditions, invalid-input handling, and scope limits.
#   May include example invocations (e.g. CLI usage).
#   Example:
#     - Add factorial as a supported calculator operation.
#     - Ensure the implementation handles valid integer inputs correctly,
#       including boundary cases such as 0 and 1, and rejects unsupported
#       inputs such as negative values or non-integers.
#     - Keep the change scoped to this feature, update the relevant tests,
#       and avoid unrelated refactoring.

# AGENT CONFIGURATION DESCRIPTIONS
# ═══════════════════════════════════════════════════════════════════════════
#
# GENERIC (single agent)
#   Claude acts as one autonomous developer. No role separation is imposed.
#   No additional behavioral constraints beyond the shared workflow prompt.
#
# TEAM (theoretical software development team)
#   Claude is instructed to simulate a full software development team by
#   working through four professional roles before committing:
#     Data Analyst    — analyse the requirements, clarify scope, identify
#                       what data/inputs/outputs the feature involves.
#     System Architect — design the solution: which modules change, how the
#                        feature fits the existing architecture, what risks exist.
#     Programmer      — implement the solution according to the architecture.
#     Tester          — write or update tests, run them, verify all pass.
#   Each role's output informs the next. Role separation is documented in
#   the PR description.
